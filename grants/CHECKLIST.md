# Grant Application Checklist — Maximize Your Chances

Use this before every submission. The more items checked, the stronger the application.

---

## Repo and code

- [ ] **Public repo** (or explicitly “will open-source upon grant acceptance” if required)
- [ ] **README** is clear: what Oak is, why Stylus, how to build/run/test
- [ ] **No broken links** in README (docs, app, scripts)
- [ ] **Tests pass**: `cargo test` (and any frontend tests)
- [ ] **License** file present (e.g. MIT)
- [ ] **Security narrative**: README or AUDIT.md mentions re-entrancy, checked math, CEI, MEV design

---

## Demo and traction

- [ ] **Testnet deployment** (Arbitrum Sepolia) — contract address in README or one-pager
- [ ] **Dashboard / app** live and linked (swap, liquidity, connect wallet)
- [ ] **One-pager** updated with real URLs (repo, app, contact)
- [ ] **Screenshots or short video** (optional but strong): “Commit → wait → reveal” or multi-hop swap

---

## Application form

- [ ] **Ask is clear**: e.g. “$20,000 for external audit and mainnet launch”
- [ ] **Use of funds** is specific: audit ($X), deployment ($Y), bootstrap/docs ($Z)
- [ ] **Milestones** are concrete and verifiable (e.g. “Public audit report”, “Live on Arbitrum One”)
- [ ] **“Why us / why this grant”** tailored to the program (Stylus, DeFi, Arbitrum ecosystem)
- [ ] **Contact** and links correct (email, Twitter, Discord, repo, app)
- [ ] **Proofread**: no placeholder text left (e.g. “[Your email]”)

---

## Narrative and differentiation

- [ ] **Problem** stated in one sentence: MEV hurts retail; parameters exposed in mempool
- [ ] **Solution** in one sentence: Commit–reveal DEX on Stylus; MEV-resistant, gas-efficient
- [ ] **First / only** angle used where true: “First MEV-protected DEX on Arbitrum Stylus”
- [ ] **Roadmap** mentioned: audit → mainnet → liquidity → integrations

---

## After submission

- [ ] **Note deadline** and any follow-up (e.g. “Decisions by date X”)
- [ ] **One follow-up** if allowed (e.g. “We’ve updated our repo with X”)
- [ ] **Apply to 2–3 more** programs with the same materials (different “why this grant” and ask if needed)

---

## Quick self-check

- Would a technical reviewer understand what Oak does in **60 seconds**? (README + one-pager)
- Would a non-technical reviewer understand **why it matters**? (MEV, fairness, Stylus)
- Is it obvious **what the grant pays for** and **what we deliver**? (audit, mainnet, transparency)

If all three are “yes,” the application is in strong shape.
